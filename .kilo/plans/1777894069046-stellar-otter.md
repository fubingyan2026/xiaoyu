### Middlewares/ .h files (4 files) — do NOT change

These headers are themselves LISTED IN `public.h`. Including `public.h` from them creates a circular self-reference (preprocessor handles it via include guards, but it is poor practice).

| File | Middlewares includes it pulls | Reason to keep as-is |
|------|-------------------------------|----------------------|
| `algorithm/line_hall_pll.h` | `filter.h` | Part of public.h itself |
| `services/can_comm/can_comm.h` | `daemon.h`, `kfifo.h`, `protocol_parser.h` | Part of public.h itself |
| `services/led/led.h` | `fsm.h`, `kfifo.h` | Part of public.h itself |
| `utils/toolkit/toolkit.h` | `memory_pool.h`, `debug.h` | Part of public.h itself |

## Edge cases

- **`perf_counter.c`** uses `<memory_pool.h>` with angle brackets — replaced with `#include "public.h"`.
- **`app.h`** has a stray `#define DEBUG_H` on line 5 (likely a buggy attempt to suppress debug.h's guard). Remove it since `debug.h` is no longer directly included.
- Each migrated file gets `#include "public.h"` placed early in the include block, after the file's own header.
- The `Middlewares/` include path (already added to CMakeLists.txt) makes `#include "public.h"` resolvable from any file.

## Risk

- **Compile-time increase**: files now transitively include all of Middlewares. Acceptable for this project size.
- **Order-dependent includes**: low risk — each Middlewares header has its own guard and includes its own dependencies.
