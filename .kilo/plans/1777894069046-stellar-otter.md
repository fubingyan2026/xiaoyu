## What I've learned from investigation

**Repository structure:**
- Two independent STM32G431 firmware projects under one monorepo:
  - `IMU_CTRL/` — FreeRTOS-based gimbal stabilization (IMU, PS4 controller, CAN bus)
  - `Q16_Drive_Opam_clion/` — FOC motor driver (field-oriented control, SVPWM, Q16 fixed-point math)
- `common/` — Shared embedded C libraries (algorithm, controller/PID, encoder, easyflash, debug, toolkit, etc.)
- Both projects duplicate common libraries into their own directory trees (IMU_CTRL references `../common/` directly via CMake; Q16_Drive has its own `Middlewares/` copy)

**Build system:**
- CMake + Ninja, `arm-none-eabi-gcc` toolchain, compile_commands.json exported for clangd
- CMakePresets.json: Debug/RelWithDebInfo/Release/MinSizeRel for both projects
- `build.sh` — convenience wrapper (auto-search, cmake configure + ninja build)
- Output: `.elf` → `.hex` + `.bin` (post-build custom command)

**Flashing:**
- IMU_CTRL: ST-Link via `STM32_Programmer_CLI`, flash address `0x08005000`
- Q16_Drive: ST-Link (`flash_stlink.sh`) or J-Link (`flash_jlink.sh`), flash address `0x08000000`
- OpenOCD: identical `stlink.cfg` in both projects

**Testing:**
- Only IMU_CTRL has tests: GoogleTest, pure algorithm code only (no HAL dependencies)
- `tests/CMakeLists.txt` fetches googletest v1.14.0 via FetchContent, tests pid.c/maths.c/filter.c from `../common/`
- Build: `cmake -B tests/build -S tests && cmake --build tests/build && ./tests/build/run_tests`

**Code style:**
- IMU_CTRL: Microsoft style (`.clang-format`), pure C17
- Q16_Drive: Google style, 2-space indent, C17 + C++17

**Toolchain quirks:**
- Compiler warnings suppressed: `-Wno-unused-function -Wno-unused-parameter -Wno-unused-variable -Wno-unused-but-set-variable`
- clangd in both projects: compilation database at `build/RelWithDebInfo`, suppresses `unused-includes` diagnostics

**Existing AGENTS.md files:**
- Subdirectory-level AGENTS.md files exist for `common/`, `Q16_FOC/`, `Core/`, `IMU_CTRL/`, and various middleware subdirs. These contain good module-level detail but don't cover the monorepo structure or cross-project coordination.
