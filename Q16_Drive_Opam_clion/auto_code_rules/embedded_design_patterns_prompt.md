【黄金法则】
在嵌入式/安全关键场景中，模式的首要目标不是"优雅"，而是：
  1. 可预测性（Determinism）   — 执行时间有界，无隐式动态行为
  2. 可追溯性（Traceability）  — 每个设计决策可追溯到需求或安全约束
  3. 可验证性（Verifiability） — 代码可被静态分析、形式化验证、单元测试覆盖
  4. 最小复杂度（Simplicity）  — 能用简单方案解决的，不引入复杂模式
  5. 如果工程中有通用模块的，优先调用 — 能有复用的模块的，不重复创建
【模式适用性评估矩阵】
每次推荐设计模式前，你须在内部评估并在输出中说明：

  | 评估维度         | 问题                                     |
  |------------------|------------------------------------------|
  | 内存分配         | 该模式是否引入堆分配？运行时是否确定？   |
  | 时间复杂度       | 最坏执行时间（WCET）是否可静态分析？     |
  | 虚函数/多态      | 是否使用虚函数？是否在硬实时路径上？     |
  | 中断安全         | 模式是否在 ISR 中使用？是否可重入？      |
  | 测试覆盖         | 是否所有分支可被单元测试覆盖？           |
  | MISRA 合规       | 是否引入 MISRA 强制规则冲突？            |
  | ASIL 等级匹配    | 模式复杂度是否与目标 ASIL 等级相称？     |

──────────────────────────────────────────────
2.1 状态管理类模式
──────────────────────────────────────────────

▶ 【有限状态机 — FSM / HSM】（首选推荐）

  适用场景：控制流程、故障管理、通信协议、模式切换
  实现要求：
  - 优先使用"表驱动 FSM"（Table-Driven），状态转移表为 const，存储于 ROM
  - 状态函数指针数组（Function Pointer Table）替代 switch-case 嵌套
  - 每个状态须有：entry_action / do_action / exit_action 三段式
  - 状态枚举须完整（不得有无名状态），default 分支须触发安全处理
  - HSM（层次状态机）使用 Miro Samek 的 QP/C 框架规范
  - 状态机须有对应的状态转移图（文字描述或 PlantUML）

  禁止：
  - 禁止多层嵌套 switch-case 实现状态机（圈复杂度失控）
  - 禁止状态函数中直接修改其他模块的全局状态（须通过事件/消息机制）

  示例骨架：
  ```c
  /* 状态处理函数原型 */
  typedef FsmStatus_t (*StateHandler_t)(FsmContext_t * const ctx_p,
                                        const FsmEvent_t   event);

  /* 状态转移表（存于 ROM） */
  static const StateHandler_t k_state_table[STATE_COUNT] = {
      [STATE_IDLE]    = Fsm_HandleIdle,
      [STATE_RUNNING] = Fsm_HandleRunning,
      [STATE_FAULT]   = Fsm_HandleFault,
  };

  /* 状态机调度（单一入口） */
  FsmStatus_t Fsm_Dispatch(FsmContext_t * const ctx_p, const FsmEvent_t event)
  {
      FSM_ASSERT(NULL != ctx_p);
      FSM_ASSERT(ctx_p->current_state < STATE_COUNT);
      return k_state_table[ctx_p->current_state](ctx_p, event);
  }
  ```

▶ 【模式寄存器（Mode Register）】

  适用场景：系统运行模式管理（正常/降级/安全/工厂模式）
  要求：模式枚举完备，模式切换须经合法性校验，禁止从任意模式跳转至任意模式

──────────────────────────────────────────────
2.2 硬件抽象类模式
──────────────────────────────────────────────

▶ 【硬件抽象层 — HAL（Hardware Abstraction Layer）】（强制要求）

  分层规范（从上到下）：
  ```
  ┌─────────────────────────────────┐
  │      Application Layer          │  业务逻辑，不含任何寄存器操作
  ├─────────────────────────────────┤
  │      Service / Driver Layer     │  设备无关驱动接口（HAL API）
  ├─────────────────────────────────┤
  │      MCU Abstraction Layer      │  芯片相关实现（MCAL）
  ├─────────────────────────────────┤
  │      Hardware Register Layer    │  寄存器映射（volatile，仅此层访问）
  └─────────────────────────────────┘
  ```
  要求：
  - HAL 接口（.h）须与实现（.c）完全解耦，宿主机（Host PC）可提供桩实现
  - 寄存器访问须封装为内联函数或宏，禁止上层直接操作寄存器地址
  - HAL 函数须返回状态码（HAL_OK / HAL_ERROR / HAL_TIMEOUT / HAL_BUSY）
  - 驱动初始化须有幂等性（重复调用不产生副作用）

▶ 【板级支持包 — BSP 抽象】

  要求：
  - 时钟、中断、DMA 配置须集中在 BSP 模块，业务代码不得直接配置
  - 引脚/外设映射须通过 board_config.h 集中定义，禁止硬编码在业务层

──────────────────────────────────────────────
2.3 内存管理类模式
──────────────────────────────────────────────

▶ 【静态内存池 — Static Memory Pool】（强制替代 malloc）

  实现要求：
  - 池大小在编译期确定（static 数组），生命周期与程序相同
  - 分配/释放时间 O(1)，无碎片
  - 须有使用率监控（高水位标记 High-Water Mark）
  - 多任务场景下须有互斥保护

  ```c
  /* 编译期确定容量的内存池 */
  #define POOL_BLOCK_SIZE   (64U)
  #define POOL_BLOCK_COUNT  (16U)

  typedef struct {
      uint8_t  data[POOL_BLOCK_SIZE];
      bool     in_use;
  } PoolBlock_t;

  static PoolBlock_t g_pool[POOL_BLOCK_COUNT];  /* 静态分配，无堆 */
  ```

▶ 【环形缓冲区 — Ring Buffer / Circular Buffer】

  适用场景：UART/CAN 收发、日志记录、数据流采集
  实现要求：
  - 索引类型须为 2^N 大小的无符号整型，使用位掩码取模（避免除法）
  - 读写指针须为 volatile（中断场景）或原子变量（多核场景）
  - 须区分"单生产者单消费者（SPSC）"和"多生产者多消费者（MPMC）"场景
  - 空/满判断须无歧义（推荐：使用计数器法或镜像位法）

▶ 【对象池 — Object Pool】

  适用场景：固定数量的消息对象、任务控制块、诊断事件
  要求：池满时须有明确的拒绝策略（丢弃/覆盖/阻塞），须文档说明

──────────────────────────────────────────────
2.4 通信与事件类模式
──────────────────────────────────────────────

▶ 【发布-订阅 — Publish-Subscribe / Event Bus】

  实现要求：
  - 订阅表为静态数组，编译期确定订阅者数量上限
  - 发布操作须为非阻塞（将事件入队列，由任务异步处理）
  - 事件 ID 须为枚举类型，须有版本控制
  - ISR 内发布事件须使用 FromISR 变体（如 FreeRTOS xQueueSendFromISR）

  禁止：
  - 禁止订阅者回调中直接执行耗时操作（违反实时性）
  - 禁止动态注册/注销订阅者（运行时修改订阅表）

▶ 【命令模式 — Command Pattern（函数指针表实现）】

  适用场景：诊断服务（UDS）、远程指令执行、参数配置
  C 语言实现：
  ```c
  typedef RetCode_t (*CmdHandler_t)(const uint8_t * const data_p,
                                     uint16_t             data_len);

  typedef struct {
      uint16_t      service_id;
      CmdHandler_t  handler;
      uint8_t       min_len;
      uint8_t       asil_level;
  } CmdEntry_t;

  static const CmdEntry_t k_cmd_table[] = {
      { 0x10U, Uds_HandleSessionCtrl,   1U, ASIL_B },
      { 0x11U, Uds_HandleEcuReset,      1U, ASIL_B },
      { 0x22U, Uds_HandleReadDataById,  3U, ASIL_A },
  };
  ```

▶ 【观察者模式 — Observer（静态注册表实现）】

  要求：
  - 观察者数组静态分配，编译期确定上限（MAX_OBSERVERS）
  - 通知顺序须确定（按注册顺序或优先级顺序）
  - 须处理观察者回调内再触发通知的递归场景（设置重入标志）

──────────────────────────────────────────────
2.5 错误处理与诊断类模式
──────────────────────────────────────────────

▶ 【错误码传播模式 — Error Code Propagation】（强制替代异常）

  实现要求：
  ```c
  /* 统一错误码类型 */
  typedef enum : uint32_t {
      ERR_OK              = 0x00000000UL,
      ERR_NULL_PTR        = 0x00000001UL,
      ERR_OUT_OF_RANGE    = 0x00000002UL,
      ERR_TIMEOUT         = 0x00000004UL,
      ERR_HW_FAULT        = 0x00000008UL,
      ERR_CHECKSUM        = 0x00000010UL,
      /* 可按位 OR 组合的复合错误 */
  } ErrCode_t;
  ```
  - 错误码须可按位组合（Bitmask），支持多错误并发上报
  - 调用链每层须决策：透传 / 转换 / 处理，不得静默丢弃
  - 关键路径错误须写入故障日志（DTC），不得只靠返回值传递

▶ 【断言策略 — Assert Strategy（分级）】

  ```c
  /* ASIL 分级断言 */
  /* 开发期断言：仅 Debug 构建生效 */
  #define DEV_ASSERT(cond)    assert(cond)

  /* 运行期安全断言：Release 构建生效，违反则进入安全状态 */
  #define SAFETY_ASSERT(cond)                          \
      do {                                             \
          if (!(cond)) {                               \
              FaultMgr_EnterSafeState(                 \
                  FAULT_ASSERT_VIOLATION,              \
                  (uint32_t)__LINE__,                  \
                  __FILE__);                           \
          }                                            \
      } while (0)

  /* 静态断言：编译期检查 */
  #define STATIC_ASSERT(cond, msg)  static_assert((cond), (msg))
  ```

▶ 【故障管理器 — Fault Manager Pattern】

  职责分离：
  ```
  故障检测层  →  故障上报层  →  故障存储层  →  故障响应层
  (Detection)    (Reporting)    (Storage/DTC)   (Response/Action)
  ```
  要求：
  - 每类故障须定义：故障码（DTC）、阈值、消抖计数、恢复条件、安全响应动作
  - 故障存储须有防写穿（Write-Through）保护，存储格式须有 CRC 校验
  - 故障响应动作须经 FMEA 分析，映射至对应 ASIL 等级的缓解措施

▶ 【看门狗喂狗模式 — Watchdog Kick Pattern】

  实现规范（任务监控矩阵）：
  ```c
  /* 每个任务维护一个生命令牌（Alive Counter） */
  typedef struct {
      uint32_t  task_id;
      uint32_t  alive_counter;
      uint32_t  expected_counter;
      uint32_t  deadline_ms;
      bool      alive_flag;
  } WdgTaskMonitor_t;
  ```
  - 看门狗监督任务（Watchdog Supervisor Task）须为最高优先级
  - 仅在所有受监控任务均报活时，才向硬件 WDG 喂狗
  - 任何任务超时均须触发系统级故障处理，禁止单独静默复位

──────────────────────────────────────────────
2.6 并发与调度类模式
──────────────────────────────────────────────

▶ 【主动对象 — Active Object（Actor 模式）】

  适用场景：封装任务 + 消息队列为独立执行单元（QP/C、FreeRTOS 实现）
  ```
  ┌──────────────────────────────────┐
  │          Active Object           │
  │  ┌────────────┐  ┌────────────┐  │
  │  │  Task/     │  │  Event     │  │
  │  │  Thread    │←─│  Queue     │  │
  │  └────────────┘  └────────────┘  │
  │  ┌────────────────────────────┐  │
  │  │       State Machine        │  │
  │  └────────────────────────────┘  │
  └──────────────────────────────────┘
  ```
  要求：
  - 每个 AO 有且仅有一个私有消息队列
  - AO 之间只能通过消息通信，禁止直接调用对方接口
  - 消息结构体须含时间戳与发送方 ID，便于调试与追溯

▶ 【保护代理 — Protected Proxy（资源访问保护）】

  适用场景：多任务共享 SPI/I2C/UART/Flash 等独占资源
  ```c
  RetCode_t Spi_Write_Protected(const uint8_t * data_p, uint16_t len)
  {
      RetCode_t ret = ERR_TIMEOUT;
      if (Mutex_Take(g_spi_mutex, SPI_MUTEX_TIMEOUT_MS) == MUTEX_OK)
      {
          ret = Spi_Write_Impl(data_p, len);  /* 实际操作 */
          Mutex_Give(g_spi_mutex);
      }
      return ret;
  }
  ```

▶ 【双缓冲 — Double Buffering】

  适用场景：传感器数据采集、显示刷新、数据快照
  要求：
  - 前台缓冲（Front Buffer）供读取，后台缓冲（Back Buffer）供写入
  - 交换（Swap）操作须为原子操作或在临界区内执行
  - 缓冲区切换须有有效性标志（Valid Flag）保护

──────────────────────────────────────────────
2.7 启动与初始化类模式
──────────────────────────────────────────────

▶ 【分阶段初始化 — Staged Initialization】

  强制初始化顺序（不得颠倒）：
  ```
  阶段 0：CPU 核心初始化（时钟、堆栈、中断向量表）
  阶段 1：硬件外设初始化（GPIO、定时器、通信外设）
  阶段 2：BSP / HAL 初始化
  阶段 3：OS 内核初始化（若使用 RTOS）
  阶段 4：安全监控模块初始化（WDG、电源监控、温度保护）
  阶段 5：应用模块初始化（按依赖顺序，依赖图须无环）
  阶段 6：自检（POST — Power-On Self Test）
  阶段 7：进入正常运行主循环
  ```
  要求：
  - 每个初始化阶段须有返回状态检查，失败须阻止进入下一阶段
  - POST 须涵盖：RAM 测试、ROM CRC 校验、关键外设通信自检
  - 初始化超时须有看门狗兜底（禁止无限等待外设就绪）

▶ 【单例 — Singleton（受限使用）】

  约束：
  - 嵌入式场景中，模块级单例须用"模块静态变量"实现，禁止使用 Meyers Singleton（依赖 C++ 局部静态初始化，线程安全性不确定）
  - 须提供 Init() / DeInit() 接口，支持单元测试中的重置
  - 须标注此模块在系统中有且仅有一个实例的依据

──────────────────────────────────────────────
2.8 参数配置类模式
──────────────────────────────────────────────

▶ 【参数管理器 — Parameter Manager / NVM Manager】

  数据分类（按存储特性）：
  ```
  ┌──────────────────────────────────────────────────────┐
  │ 类型          │ 存储位置 │ 可写性    │ 保护机制       │
  │───────────────│──────────│───────────│────────────────│
  │ 只读常量      │ Flash/ROM│ 不可写    │ 编译期 CRC     │
  │ 出厂标定值    │ NVM      │ 仅工厂    │ 双备份 + CRC   │
  │ 用户可配参数  │ NVM      │ 受权限控制│ CRC + 范围检查 │
  │ 运行时动态值  │ RAM      │ 程序可写  │ 互斥保护       │
  └──────────────────────────────────────────────────────┘
  ```
  - NVM 参数须有版本号，支持升级时的迁移（Migration）机制
  - 参数写入须有磨损均衡（Wear Leveling）或足够的写入次数余量
  - 所有参数须有默认值（Default Value），初始化时若 CRC 失败须恢复默认值

第三部分：禁用/限制模式清单

【严格禁止在安全关键场景使用的模式】

| 模式               | 禁止原因                                   | 合规替代方案                     |
|--------------------|--------------------------------------------|----------------------------------|
| 动态工厂模式       | 运行时动态分配对象，WCET 不确定            | 静态对象池 + 枚举索引            |
| 原型模式（Clone）  | 深拷贝涉及动态内存，生命周期不明确         | 显式初始化函数                   |
| 模板方法（虚函数） | 硬实时路径上虚函数调度延迟不确定           | 函数指针表（编译期绑定）         |
| 装饰器模式         | 多层包装导致调用链深度不可预测             | 组合配置结构体                   |
| 责任链模式         | 动态链表结构，遍历 WCET 不确定             | 优先级固定的静态处理数组         |
| 解释器模式         | 运行时解析复杂度高，安全性难以验证         | 预编译状态机或查找表             |
| 代理模式（动态）   | 运行时决定委托目标，控制流不透明           | 编译期绑定的保护代理             |
| 享元模式           | 共享内部状态引入隐式耦合，多任务下危险     | 显式参数传递                     |

【有条件使用的模式（须满足约束）】

| 模式         | 约束条件                                                       |
|--------------|----------------------------------------------------------------|
| 策略模式     | 策略对象须静态分配；策略切换须在任务上下文，不得在 ISR 中切换  |
| 模板模式     | 仅用 C++ 模板（编译期多态），禁止虚函数；须评估代码膨胀影响   |
| 建造者模式   | 构建过程须完全在初始化阶段完成，禁止运行时增量构建             |
| 迭代器模式   | 仅用于静态容器，迭代器生命周期须不超过容器生命周期             |

```
┌─────────────────────────────────────────────────────┐
│             提示词使用分工建议                        │
├───────────────────┬─────────────────────────────────┤
